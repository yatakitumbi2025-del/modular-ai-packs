# Modular AI — Domain Pack Format

This repo defines the **contract** for domain packs used by a modular, on-device AI.
The language model runs **locally** on the user's device. Domain packs live **online**
in this repo and are fetched on demand — only the pack the user activates is downloaded.

## The two-file contract

- **`registry/index.json`** — the registry. The app fetches this first. It lists every
  available pack with just enough metadata to show a pick-list and route: `id`, `name`,
  `description`, `version`, `tags`, `size_bytes`, and the path to each pack's manifest.
  It also declares the **one embedding model** every pack is standardized against.

- **`packs/<id>/pack.json`** — a pack manifest. Downloaded only when that pack is picked.
  It points to the pack's other files (prompt, examples, knowledge, vectors) and carries
  routing signals.

`schema/pack.schema.json` formally validates any `pack.json`.

## What a pack folder contains

```
packs/programming/
├── pack.json                 # manifest (points to everything below)
├── prompt.md                 # the module's persona / system prompt
├── examples.md               # few-shot examples
├── knowledge/                # source documents (markdown) for RAG
│   └── python-comprehensions.md
├── vectors.json              # precomputed embeddings of the knowledge/ chunks
└── (tools declared in pack.json, e.g. a code_runner)
```

Text files are tiny and load instantly. `vectors.json` is the only heavier piece, and it
downloads only when the domain is activated.

## How the app uses this at runtime

1. Fetch `registry/index.json`, show the user the list of domains.
2. **Route:** given a user question, pick the domain(s). Cheap path = match against each
   pack's `routing.keywords`. Better path = embed the question and compare to each pack's
   `routing.description_for_router`.
3. Fetch the chosen `pack.json`, then its `prompt.md` + `examples.md`.
4. **RAG:** embed the user's question on-device with the registry's embedding model,
   cosine-compare against the vectors in `vectors.json`, take the top few chunks.
5. Build the prompt: `prompt.md` + `examples.md` + retrieved chunks + the question.
   Send it to the local model.
6. If multiple domains were needed, merge their outputs and run a self-check.

## Serving

Push this repo to GitHub, then fetch via the **jsDelivr** CDN (free, adds CORS headers,
globally cached), e.g.:

```
https://cdn.jsdelivr.net/gh/<user>/<repo>/registry/index.json
```

## Adding a new pack

1. Create `packs/<id>/` with `pack.json`, `prompt.md`, `examples.md`, and optional
   `knowledge/` + `vectors.json`.
2. Validate `pack.json` against `schema/pack.schema.json`.
3. Add a one-line entry to `registry/index.json`.
4. Open a pull request.

## Generating `vectors.json` (important)

The `vectors.json` shipped in the programming pack contains **placeholder** vectors (8 values
each) so you can see the shape. Real vectors have 384 floats and must be generated by a
build script that: (a) splits each `knowledge/*.md` into chunks, (b) embeds each chunk with
the **same** embedding model named in the registry (`all-MiniLM-L6-v2`), (c) writes out the
`{ id, source, text, vector }` records. Query-time embedding must use that identical model,
or the cosine comparison is meaningless.
